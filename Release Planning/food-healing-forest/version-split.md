# 食愈森林版本切分方案

## 1. 推荐版本方案

### 1.1 MVP
- 目标：验证 `内容拉起 -> 发起记录 -> 饭后补填 -> 轻反馈` 是否能形成最小留存闭环。

#### MVP 必做
- `每日打开入口`
  - 日签内容
  - 首页主记录入口

- `一餐记录发起`
  - 拍照/上传
  - 选择当下心情
  - 创建记录

- `饭后补感受闭环`
  - 默认饭后提醒
  - 补充身体感受
  - 完整记录状态更新

- `记录回看与完成反馈`
  - 当天与历史记录查看
  - 完成记录后的轻反馈

- `森林基础成长反馈`
  - 连续记录天数
  - 单植物基础成长状态

- `轻运营后台基础能力`
  - 日签内容生成与审核
  - 基础标签和提醒规则
  - 记录发起、完整记录、回填率、内容转化看板

#### MVP 明确不做
- 一分钟正念
- 今日宜食
- 场景食方库
- 月度洞察
- 树洞信箱
- 食材红绿灯百科

### 1.2 V1
- 目标：在主闭环跑通后，增强打开理由和周期回访价值。

#### V1 建议加入
- 一分钟正念
- 今日宜食
- 场景食方首批内容
- 月度轻洞察
- 后台扩展支持宜食/食方/月报文案生成与审核

#### 为什么放在 V1
- 这些能力能增强打开频次和品牌记忆，但前提是记录闭环已经成立。
- 如果主闭环尚未稳定，先加内容深度只会增加复杂度，不一定提升留存。

### 1.3 V2
- 目标：增强差异化和长期陪伴感。

#### V2 建议加入
- 食材红绿灯轻百科
- 树洞信箱
- 更丰富的森林成长层
- 更精致的日签分享样式

#### 为什么放在 V2
- 这些能力要么依赖较强风控，要么更偏品牌和内容深度，不适合首阶段抢资源。

### 1.4 Backlog
- 个性化推荐
- 多植物、多地块复杂养成
- 精细内容运营活动
- 更复杂的图像滤镜、贴纸、分享编辑器

## 2. 版本顺序逻辑

### 2.1 先做什么
- 先打通记录闭环。
- 先验证饭后补感受这一关键动作是否真的可持续。

### 2.2 再做什么
- 再补充更多让用户愿意每天来的理由。
- 再强化周期洞察和内容场景丰富度。

### 2.3 最后做什么
- 最后再做高风控、高内容深度、强品牌差异化模块。

## 3. 关键取舍说明

### 3.1 为什么 MVP 只保留日签，不把首页全做满
- 因为首版首页的任务不是“多内容承载”，而是“提供一个进入记录的轻理由”。
- 日签已经足够承担这个作用，继续加入正念和宜食会增加内容供给压力和设计复杂度。

### 3.2 为什么森林只保留基础成长
- 因为首版森林的职责是反馈，不是玩法。
- 过早强化游戏化，会让产品从“自我觉察”偏向“任务打卡”。

### 3.3 为什么月度洞察不进 MVP
- 因为它依赖数据积累。
- 如果记录量不足，月度洞察的感知价值不高，还会稀释主链路资源。

### 3.4 为什么树洞信箱后移
- 因为这项能力的风险和审核成本明显高于首版其他功能。
- 它属于可以增强陪伴感的模块，但不属于验证记录闭环的必要条件。

## 4. 约束敏感点

- 如果小程序提醒能力受限：
  - MVP 仍可上线，但 `饭后补感受闭环` 的效果会被削弱。
  - 这会让完整记录率和连续记录率的解释力下降。

- 如果 AI 内容风格不稳定：
  - 首版应收缩到少量内容类型，保证审核质量优先。

- 如果后台资源有限：
  - 必须保住日签、记录、提醒、看板这四类能力。
  - 其余内容能力继续顺延。

## 5. 推荐结论

- 推荐版本切分：`MVP = 记录闭环验证版`，`V1 = 内容和洞察增强版`，`V2 = 差异化和深陪伴版`
- 这是当前最符合项目目标和资源现实的切法。
