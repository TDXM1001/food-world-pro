# 食愈森林 M0 阻塞项 1：饭后提醒能力决策稿

## 1. 文档目标

- 为 `M0 阻塞项确认会` 提供一份可拍板的提醒能力决策稿。
- 回答 3 个问题：
  - MVP 是否可以把“饭后 2 小时提醒”作为刚性闭环依赖。
  - 微信小程序当前官方能力下，什么提醒路径是稳的，什么路径不能默认假设成立。
  - 产品、研发、测试应按什么口径落地，才能既不失去回补率，又不把 MVP 建在不确定能力上。

## 2. 当前决策问题

- 当前 PRD 中，`PRD-006` 和 `PRD-016` 都依赖“饭后提醒”能力。
- 但对微信小程序而言，提醒不是一个可以无条件使用、无条件触达、无条件横幅通知的能力。
- 因此本次会议需要先拍板：
  - `MVP 的完整记录闭环，是否必须依赖外部提醒`
  - `是否将订阅消息作为增强项，而不是主闭环唯一前提`

## 3. 官方能力快照

> 以下内容基于 2026 年 4 月 23 日核对的微信开放文档。

### 3.1 已确认的官方事实

- 小程序订阅消息开发指南：
  - 官方文档：<https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/subscribe-message.html>
  - 文档确认存在订阅消息能力、语音提醒能力和“添加提醒”能力。
- `wx.requestSubscribeMessage`：
  - 官方文档：<https://developers.weixin.qq.com/miniprogram/dev/api/open-api/subscribe-message/wx.requestSubscribeMessage.html>
  - 基础库 `2.4.4` 开始支持。
  - `2.8.2` 起，用户发生点击行为或者发起支付回调后，才可以调起订阅消息界面。
  - 一次性模板 id 和永久模板 id 不可同时使用。
  - 一次调用最多可订阅 5 条消息。
  - 成功回调会返回每个模板的结果，包含 `accept / reject / ban / filter`。
  - 错误码 `20004` 表示用户关闭了主开关，无法进行订阅。
- 服务端发送订阅消息：
  - 官方接口列表：<https://developers.weixin.qq.com/miniprogram/dev/server/API/>
  - 文档确认存在 `/cgi-bin/message/subscribe/send` 接口，用于发送订阅消息。
- 微信内“添加提醒”能力：
  - 官方指南确认：用户在订阅小程序 `长期订阅消息` 时，可以根据自己的使用情况添加提醒。
  - 添加后，用户收到消息时会在微信内由横幅通知提醒。
  - 该状态可通过 `wx.getSetting` 读取为 `acceptWithAlert`。

### 3.2 可以明确推出的结论

- 结论 A：订阅弹窗不是系统任意时刻可拉起的，它依赖用户点击时机。
- 结论 B：即使我们做了订阅消息，也不能把“每条记录都一定能成功授权提醒”当作默认事实。
- 结论 C：微信内横幅级提醒能力，官方明确挂在 `长期订阅消息` 上，不能默认套用到所有提醒方案。

### 3.3 当前仍未确认的点

- 食愈森林所在类目是否具备可用的长期订阅模板能力。
- 如果不能拿到长期订阅能力，我们是否只能使用一次性订阅消息。
- 目标人群对订阅授权的接受率是否足以支撑完整记录率目标。

## 4. 决策选项

## 4.1 方案 A：订阅消息主方案

### 方案定义
- 在用户提交餐记录后，主打“2 小时后提醒我”。
- 将订阅消息视为完整记录闭环的主路径。
- 产品增长目标里，默认较高依赖提醒触达和提醒回补。

### 优点
- 用户感知最直接，最符合“饭后回来补感受”的原始设想。
- 如果授权和触达都稳定，回补率理论上更高。
- 后台提醒策略更容易做精细化优化。

### 缺点
- 过度依赖平台触达能力。
- 依赖用户在点击场景下完成授权，且不能保证所有用户都会接受。
- 如果项目无法稳定使用长期订阅能力，就很难承诺微信内横幅提醒体验。
- 一旦提醒失败，MVP 核心闭环会被连带打断。

### 对执行的影响
- 产品：需要把授权、拒绝、主开关关闭、模板不可用等分支全部写清。
- 研发：需要前端授权、服务端发送、任务调度、失败回退全链路。
- 测试：要覆盖授权态、关闭主开关、模板问题、发送失败等异常。
- 运营：需要持续维护模板和提醒文案。

### 适用前提
- 已确认项目可稳定使用合适的订阅模板。
- 已确认提醒链路是 MVP 主路径，团队愿意为其承担实现和测试复杂度。

### 当前判断
- `不推荐作为 MVP 默认主方案`

## 4.2 方案 B：手动回补主方案

### 方案定义
- 不把外部提醒作为 MVP 必要前提。
- 用户提交餐记录后，系统在手帐历史、首页和森林里持续展示“待补感受”。
- 用户通过主动回看和产品内提示完成回补。

### 优点
- 能力稳定，不依赖平台提醒触达。
- 研发和测试复杂度最低。
- 可以更快落 MVP，闭环定义更稳。

### 缺点
- 回补率上限可能偏低。
- 用户需要主动返回，行为成本更高。
- “饭后回来看看”的心智需要靠产品内提示慢慢建立。

### 对执行的影响
- 产品：需要强化产品内待办提示和历史回补入口。
- 研发：主要是状态展示、待补入口、回补流程。
- 测试：聚焦状态流转与手动回补，不用覆盖大量提醒异常。
- 运营：几乎不需要维护提醒策略。

### 适用前提
- 团队希望 MVP 先验证“有人愿意完整记录”，而不是先验证“外部提醒是否能打通”。

### 当前判断
- `可作为最低风险保底方案`

## 4.3 方案 C：混合保底方案

### 方案定义
- 把“手动回补”作为 MVP 保底闭环。
- 把“订阅消息提醒”作为可选增强路径，而不是唯一闭环前提。
- 用户在提交记录后，可点击“2 小时后提醒我”触发订阅授权。
- 授权成功则尝试走提醒链路；授权失败、主开关关闭或模板不可用时，自动退回手动回补路径。

### 优点
- 同时保住“闭环稳定性”和“提醒增强空间”。
- 不会因为提醒能力不确定，就让完整记录率目标失真。
- 与当前 PRD 中的 `reminder_status` 分层设计天然匹配。
- 后续如果确认长期订阅能力成立，可以在不推翻 MVP 的前提下增强提醒体验。

### 缺点
- 比纯手动回补多一层实现复杂度。
- 指标拆分会更复杂，需要区分“已订阅用户回补率”和“整体回补率”。
- 页面上需要处理“提醒成功”和“提醒不可用”的差异文案。

### 对执行的影响
- 产品：
  - 闭环定义改为“提醒增强，不是闭环前提”。
  - 记录完成率不能只归因于提醒触达。
- 研发：
  - 业务状态仍以 `awaiting_post_meal_feedback -> completed` 为主。
  - 提醒状态独立维护，失败时写入 `not_scheduled`。
- 测试：
  - 覆盖两条路径：提醒成功回补、提醒失败后手动回补。
- 运营：
  - 可逐步验证订阅文案、授权率、发送率、回补率，而不用首版就压重策略复杂度。

### 适用前提
- 团队接受 MVP 首版中“提醒能力是增强项而不是产品承诺”。

### 当前判断
- `推荐方案`

## 5. 推荐结论

### 5.1 推荐选项
- 推荐采用 `方案 C：混合保底方案`

### 5.2 推荐原因
- 食愈森林当前最重要的不是“把提醒系统做完整”，而是“确保完整记录闭环可成立”。
- 微信官方能力已经明确：
  - 订阅弹窗依赖用户点击触发。
  - 微信内横幅提醒能力明确挂在长期订阅消息上。
- 在长期订阅模板资格尚未确认前，把提醒当作唯一闭环前提风险过高。
- 混合保底方案既保留了提醒带来的增长空间，又不会让 MVP 建在不稳定能力上。

## 6. 建议拍板口径

### 6.1 M0 建议决策
- `决策 1`：MVP 不把外部提醒作为完整记录闭环的唯一依赖。
- `决策 2`：MVP 若接入订阅消息，仅作为增强路径，不改变主业务状态定义。
- `决策 3`：在长期订阅资格未确认前，不对外承诺“微信内固定横幅提醒”体验。
- `决策 4`：PRD、接口、埋点、测试全部按“手动回补保底 + 订阅消息增强”口径执行。

### 6.2 对 PRD 的同步影响
- `PRD-006`：
  - 保留默认尝试安排 2 小时提醒。
  - 明确提醒失败不影响记录继续手动回补。
- `PRD-016`：
  - 重点从“强提醒系统”改为“提醒增强能力与状态追踪”。
- `PRD-017`：
  - 看板需拆开看：
    - 授权率
    - 提醒发送率
    - 提醒回补率
    - 整体完整记录率

## 7. MVP 最小实现建议

- 提交餐记录成功后，展示一个按钮：`2 小时后提醒我`
- 仅在用户点击按钮时，调用 `wx.requestSubscribeMessage`
- 授权成功：
  - 创建提醒任务
  - `reminder_status = scheduled`
- 授权拒绝、主开关关闭、模板不可用、发送失败：
  - `reminder_status = not_scheduled`
  - 页面仍提示“稍后可在手帐里补感受”
- 手帐页、首页和森林页都要对 `awaiting_post_meal_feedback` 记录给出回补入口
- 不在首版文案中承诺“必达提醒”或“固定横幅提醒”

## 8. 会上要拍的 4 个问题

1. MVP 是否接受“手动回补是保底闭环”这一定义。
2. MVP 是否接入一次性订阅消息增强链路。
3. 长期订阅能力是否作为后续确认项，而不是当前开发前置条件。
4. 指标上是否将“提醒回补率”和“整体完整记录率”拆开看。

## 9. 当前建议

- 如果团队目标是“最快验证完整记录闭环”，优先选 `方案 B` 或 `方案 C`。
- 如果团队目标是“首版就强打提醒回填”，才考虑 `方案 A`，但前提是先确认平台资格和实现资源。
- 在当前信息下，最稳妥的路径仍然是：`方案 C：混合保底方案`。
