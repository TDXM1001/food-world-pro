# 食愈森林 后端任务拆分前置清单

## 1. 文档目标

- 判断当前是否已经具备“开始正式拆后端开发任务”的条件。
- 把“可以预研”和“可以拆正式任务”明确区分，避免后端过早按不稳定口径开工。
- 作为后端正式接包前的最后一道内部判断清单。

## 2. 当前建议口径

- 当前默认判断：`还没到正式拆后端任务`
- 当前允许动作：
  - 数据模型草案
  - 接口草案
  - 异步任务方案草案
  - 权限与风控方案草案
- 当前不建议动作：
  - 直接拆正式研发任务
  - 锁死表结构与状态字段
  - 承诺联调排期

## 3. 前置清单

## A. 会议结论关口

- `M0` 是否已有正式会议结论记录
- `M1` 是否已有正式评审结论记录
- `M0 / M1` 结论是否已经完成会后回写

## B. 范围关口

- 后台范围是否已明确冻结为 `ADM-01 ~ ADM-05`
- 是否已经明确本轮不做项，不再进入后端排期讨论
- 是否不存在“先按完整四入口产品形态做着看”的隐性理解

## C. 状态与字段关口

- `business_status` 是否已最终冻结
- `reminder_status` 是否已最终冻结
- `MealRecord / ReminderTask / DashboardSummary / SensitiveCase` 是否已可进入正式 API 文档
- 是否仍残留会影响实现方式的 `待 M0 确认` 字段

## D. 权限与删除关口

- 敏感内容默认不可见范围是否已明确
- 后台查看原始敏感内容的角色是否已明确
- 删除策略是否已明确为前台自助或后台人工兜底
- 若为人工兜底，SLA 是否明确

## E. 接口映射关口

- 页面到接口映射是否完整
- 后台首页、日签审核、规则维护、看板、敏感处置是否都有对象和接口占位
- 是否不存在页面要数据、后端却还不知道字段从哪来的情况

## F. 项目节奏关口

- 是否已经过开发移交前检查
- 是否已经准备好开发移交会材料
- 是否已经明确后端依赖谁先给输入、谁后接联调

## 4. 判定标准

### 4.1 `可以开始正式拆后端任务`

- 上述关口全部通过，或仅剩不会改变实现方式的细节备注
- 后端可以开始拆接口开发、任务调度、管理后台、数据聚合等正式研发任务

### 4.2 `只能预研，不能正式拆任务`

- 核心范围已经大致清楚
- 但仍有状态、权限、删除、提醒链路等关键口径未锁死
- 这时后端只能产出草案，不应进入正式工时承诺

### 4.3 `不建议启动后端实现`

- `M0` 或 `M1` 还没有正式结论
- 页面和接口边界仍在漂移
- 存在会直接改动表结构或任务链路的未决项

## 5. 当前食愈森林的建议判断

- 当前判断：`只能预研，不能正式拆后端任务`
- 原因：
  - 你们已经有了后端接包范围和接口字段占位
  - 但还缺 `M0` 和 `M1` 的正式会审结论与回写完成态
  - 所以离“后端正式接包开发”还差最后两道流程关口

## 6. 会后建议动作

1. 先完成 `M0` 正式结论并回写。
2. 再完成 `M1` 正式评审并回写。
3. 然后用 [dev-handoff-readiness-checklist.md](</e:/my-project/food-world-pro/Execution Alignment/food-healing-forest/dev-handoff-readiness-checklist.md>) 做正式移交判断。
4. 通过后，再进入后端任务拆分和排期。

## 7. 当前建议

- 你现在问“是不是还没到后端那一步”，答案是：`对，还没到正式开发那一步，但已经到了后端预研与接包准备这一步。`
- 所以下一阶段不是继续补大而全的方案，而是把 `M0` 和 `M1` 走完，拿到能支撑后端正式拆任务的结论。
- 如需最短推进顺序，可直接按 [m0-m1-shortest-execution-order.md](</e:/my-project/food-world-pro/Execution Alignment/food-healing-forest/m0-m1-shortest-execution-order.md>) 执行。
