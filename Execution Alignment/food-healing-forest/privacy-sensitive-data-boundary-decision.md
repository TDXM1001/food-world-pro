# 食愈森林 M0 阻塞项 3：隐私与敏感数据边界决策稿

## 1. 文档目标

- 为 `M0 阻塞项确认会` 提供一份可拍板的隐私与敏感数据边界决策稿。
- 本文件回答 5 个问题：
  - 食愈森林 MVP 到底处理哪些用户数据。
  - 哪些数据应按高敏感口径处理，不能和普通运营数据混用。
  - 后台哪些角色可以看什么，哪些角色默认不能看。
  - 日志、看板、AI 生成、风控是否可以直接使用原始数据。
  - MVP 如果暂不做前台自助删除，最低可接受的替代方案是什么。

> 本文件是产品与执行口径，不替代正式法律意见。上线前仍需由法务或合规角色做最终确认。

## 2. 当前决策问题

- 当前 PRD 已明确 MVP 会处理：
  - 餐图
  - 当下心情标签
  - 饭后身体感受标签
  - 记录状态、提醒状态、连续记录数据
- 当前 PRD 也预留了未来可能处理：
  - 树洞自由文本
  - 月报文案生成
  - 风控命中记录
- 但团队尚未统一：
  - 这些数据按什么敏感等级管理
  - 后台哪些角色可以查看原始数据
  - 是否允许日志和 BI 直接落原始内容
  - 删除与停用策略的最低要求

## 3. 当前涉及的数据范围

### 3.1 MVP 已在范围内的数据

- `餐图`
- `meal_time`
- `mood_tags`
- `post_meal_feeling_tags`
- `business_status`
- `reminder_status`
- `streak / forest_growth_applied / completed_at`
- 记录相关埋点与错误日志

### 3.2 未来版本可能涉及的数据

- `文字备注`
- `树洞自由文本`
- 树洞风险分级结果
- 更丰富的月报洞察输入数据

## 4. 推荐数据分级

> 这里的“分级”是食愈森林项目内的执行分级，用于决定可见范围、存储方式和日志策略，不等同于法律条文中的正式分级术语。

### 4.1 A 类：低敏业务数据

- 内容模板 ID
- 内容发布时间
- 规则配置项
- 聚合后的看板指标
- 非用户级的运营统计

### 4.2 B 类：个人行为数据

- `record_id`
- `user_id`
- `meal_time`
- `business_status`
- `reminder_status`
- 完成时间、连续记录天数
- 埋点事件中的页面来源、按钮点击结果

### 4.3 C 类：高敏感自我感受数据

- `餐图`
- `mood_tags`
- `post_meal_feeling_tags`
- 用户后续若补充的文字备注

### 4.4 D 类：高风险自由表达数据

- 树洞自由文本
- 树洞风险标签
- 任何可直接表达身材焦虑、饮食失调、自伤倾向等内容的用户原文

## 5. 决策选项

## 5.1 方案 A：宽松运营可见方案

### 方案定义
- 后台默认允许产品、运营、客服查看用户的原始记录数据。
- 原始餐图、情绪标签、身体感受标签可直接进入常规后台页面。
- 看板支持按用户级下钻查看原始内容。

### 优点
- 排查问题最直接。
- 运营和客服理解用户样本更方便。
- 研发初期实现成本最低。

### 缺点
- 与“低焦虑、私密陪伴”的产品定位冲突。
- 原始餐图和情绪/身体感受数据暴露范围过大。
- 后续一旦做树洞，自由文本风险会被成倍放大。
- 容易导致日志、看板、导表习惯失控。

### 当前判断
- `不推荐`

## 5.2 方案 B：有限白名单方案

### 方案定义
- 后台默认只看用户级元数据。
- 少数白名单角色可查看原始餐图与标签，用于申诉、故障排查或风险复核。
- 树洞若未来上线，仍需要额外独立权限。

### 优点
- 比宽松方案安全。
- 仍保留人工排查和客服处理的空间。
- 对研发改造量相对适中。

### 缺点
- 如果白名单口径不严，很容易逐步扩张。
- 仍需设计较细的权限、导出、审计规则。
- 如果没有明确使用条件，白名单会变成事实上的默认可见。

### 当前判断
- `可作为过渡方案，但需要强约束`

## 5.3 方案 C：最小必要 + 角色隔离方案

### 方案定义
- 将 `餐图 / 情绪标签 / 身体感受 / 树洞原文` 全部视为高敏感用户内容。
- 默认后台不向产品、内容运营、普通客服开放原始内容浏览能力。
- 看板只展示聚合指标，不展示用户原始内容。
- 原始数据仅在以下场景受控访问：
  - 用户明确发起问题反馈或申诉
  - 研发排查指定记录故障
  - 风控或合规处理命中事件
- 树洞原文若未来上线，必须独立存储、独立权限、独立风控。

### 优点
- 最符合产品的“轻陪伴、低暴露、低焦虑”定位。
- 把未来树洞的风险提前隔离，不会到 V2 再返工。
- 有利于控制日志、导出、BI 下钻蔓延。
- 更适合后续对外说明数据处理边界。

### 缺点
- 权限和排障流程设计更复杂。
- 客服和运营不能再靠“直接看用户内容”做日常判断。
- 需要明确异常排查申请机制。

### 当前判断
- `推荐方案`

## 6. 推荐结论

### 6.1 推荐选项
- 推荐采用 `方案 C：最小必要 + 角色隔离方案`

### 6.2 推荐原因

- 食愈森林的核心体验不是“效率型记录工具”，而是“低压力自我觉察工具”。
- 用户上传的不是普通业务附件，而是：
  - 饮食图像
  - 情绪标签
  - 身体感受标签
- 这些内容一旦在后台被广泛可见，会直接伤害产品信任感。
- 未来树洞一旦引入自由文本，如果不提前定义隔离边界，MVP 之后的权限体系会被整体推翻。

## 7. 建议拍板口径

### 7.1 数据分级口径

- `餐图 / 情绪标签 / 身体感受标签` 按项目内 `高敏感用户内容` 处理。
- `树洞自由文本` 按项目内 `高风险自由表达内容` 处理。
- `记录状态 / 提醒状态 / 完成时间 / 连续记录` 可作为个人行为数据处理，但不应与原始高敏感内容混用。

### 7.2 存储边界

- 原始餐图只用于：
  - 用户记录回看
  - 同一用户的月报或森林反馈生成
  - 指定问题排查
- `mood_tags` 与 `post_meal_feeling_tags` 只用于：
  - 记录展示
  - 同一用户的洞察归纳
  - 聚合统计
- 不将原始餐图、树洞原文、原始情绪/感受内容默认用于通用模型训练。
- 不把原始树洞内容混入普通 BI 明细表。

### 7.3 后台可见范围

- 内容运营：
  - 只看内容生成、审核、发布相关数据
  - 不看用户原始记录
- 数据分析：
  - 只看聚合指标和匿名化趋势
  - 不看用户原始餐图和树洞原文
- 普通客服：
  - 默认只看记录状态、时间、是否完成、错误码
  - 不默认看原始餐图和高敏感标签
- 风控 / 合规 / 指定技术支持：
  - 在有明确工单或命中事件前提下，受控查看原始内容
- 所有原始内容查看动作：
  - 应有访问记录或审计日志

### 7.4 日志与看板口径

- 应记录：
  - 上传成功/失败
  - 提醒创建结果
  - 风控命中结果
  - AI 生成结果
- 不应记录到普通日志的内容：
  - 原始餐图
  - 树洞原文
  - 可直接还原用户表达的完整文本
- 看板应优先展示：
  - 完整记录率
  - 提醒回补率
  - 场景内容转化率
  - 风控命中率
- 看板不应默认支持：
  - 按用户下钻查看原始餐图
  - 按用户下钻查看树洞原文

### 7.5 删除与停用策略

- 最理想口径：
  - MVP 支持前台单条记录删除
- 最低可接受口径：
  - 即使首版前台不开放删除，也必须在上线前具备后台人工删除流程
  - 删除后记录应立即从前台隐藏
  - 删除后该记录不得再参与月报、森林成长和聚合分析
- 对树洞未来版本的预埋口径：
  - 用户应具备删除或清空能力
  - 树洞删除不应只做“前台不可见”，还需同步停用后续生成与分析

## 8. MVP 最小实现建议

### 8.1 前台提示

- 首次进入记录链路时，给出简洁说明：
  - 为什么需要餐图
  - 为什么需要心情和身体感受标签
  - 这些信息用于什么，不用于什么
- 提示重点应是：
  - 用于记录回看
  - 用于形成自己的轻洞察
  - 不做公开展示

### 8.2 服务端与后台

- 将用户记录数据和内容运营数据分表或分模块隔离。
- 原始餐图 URL 不进入普通 BI 明细表。
- 高敏感标签在日志里仅记录代码值或统计结果，不直接落展示文本。
- 风控命中结果和用户原文分开存储。

### 8.3 树洞预埋边界

- 虽然树洞不在 MVP 主范围内，但现在就要锁口径：
  - 树洞原文不得进入常规运营后台
  - 树洞原文不得用于内容训练
  - 树洞必须有独立权限与风险处理链路

## 9. 会上要拍的 5 个问题

1. `餐图 / 情绪 / 身体感受` 是否统一按高敏感用户内容处理。
2. MVP 首版是否开放前台单条记录删除；若不开放，后台人工删除 SLA 如何定义。
3. 后台哪些角色可以查看原始餐图，是否需要白名单和访问审计。
4. BI 看板是否只保留聚合指标，不开放原始内容下钻。
5. 树洞虽然延期，但是否现在就确认“独立存储、独立权限、不得入普通分析”的边界。

## 10. 当前建议

- 如果团队想尽快推进 MVP，又不想在后面返工权限体系，最稳的是现在就拍板 `方案 C`。
- 如果这次会议只能锁一个最低前提，那就先锁：
  - `原始高敏感内容默认不进入普通后台可见范围`
- 如果删除能力暂时不做前台自助，也不要直接跳过，至少要在开发前锁定后台人工删除方案。
