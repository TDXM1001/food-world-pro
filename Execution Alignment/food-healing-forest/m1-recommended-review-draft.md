# 食愈森林 M1 评审参考样稿

## 1. 使用说明

- 本文档用于 `M1 设计与接口冻结评审` 前的建议结论准备，不替代正式评审记录。
- 作用是让参会人先看到“建议通过什么、保留什么、哪些不能再争论”，提高评审效率。
- 正式评审结论仍以 [m1-review-record-template.md](</e:/my-project/food-world-pro/Execution Alignment/food-healing-forest/m1-review-record-template.md>) 回写版本为准。

## 2. 建议前提

- 默认 `M0` 已按推荐口径完成拍板。
- 当前评审只看 `MVP`，不把 `V1 / V2` 内容模块混入本轮冻结范围。
- 当前评审重点是“结构和边界”，不是高保真视觉喜好。

## 3. 推荐会议总结论

- 总体结论：`Conditional Go`
- 是否允许进入 `开发移交前检查`：`允许，但需先完成 M1 回写`
- 条件说明：
  - 评审通过后，必须先完成 `原型 / 页面清单 / 字段占位 / 执行文档` 同步。
  - 未同步前，只允许研发继续预研，不建议直接宣布正式接包开发。

## 4. 评审项建议结论

### 4.1 页面范围结论

- 当前结论：`Go`
- 建议通过页面：
  - 小程序：`MP-01 ~ MP-06`
  - 后台：`ADM-01 ~ ADM-05`
- 建议不进入本轮页面：
  - 食方首页
  - 场景食方详情
  - 一分钟正念全屏页
  - 今日宜食详情
  - 月度洞察页
  - 树洞信箱
  - 食材红绿灯百科
- 会议建议口径：
  - 本轮冻结只围绕“晨光拉起 -> 一餐记录 -> 饭后补感受 -> 历史回看 -> 森林反馈 -> 后台支撑”闭环。
  - 任何新增页面如不能直接增强这条闭环，默认不进入 `M1`。

### 4.2 页面结构结论

- 当前结论：`Conditional Go`
- 小程序结构建议：
  - `MVP` 采用 `晨光 / 手帐 / 森林` 三入口结构。
  - `食方` 在本轮保持为后续版本能力，不进入当前底部主导航。
  - `MP-04 手帐列表` 作为“待补记录回流入口”保留核心优先级。
- 后台结构建议：
  - 后台保持“总览 / 日签审核 / 规则维护 / 转化看板 / 敏感处置”最小能力结构。
  - 不引入额外复杂运营工作台。
- 本项之所以是 `Conditional Go`：
  - 需要明确团队是否完全接受 `MVP` 三入口，而不是再按完整产品愿景理解成四入口。

### 4.3 状态与流程结论

- 当前结论：`Go`
- 建议口径：
  - 首页发起记录链路唯一。
  - 饭后补感受支持从提醒和手帐列表两条路径回流。
  - `business_status` 与 `reminder_status` 继续保持分层，不在页面结构层重新混合。
  - 森林成长只在完整记录达成后触发，不在“只上传图片”阶段提前触发。
- 需在评审中再次确认的点：
  - `MP-04` 中“待补”与“已完成”的分组展示方式
  - 删除后的列表展示和统计口径

### 4.4 字段与接口占位结论

- 当前结论：`Conditional Go`
- 已建议通过的对象：
  - `DailyCard`
  - `MealRecord`
  - `ReminderTask`
  - `ForestSummary`
  - `TagOption`
  - `DashboardSummary`
  - `SensitiveCase`
- 建议口径：
  - 当前字段占位已足够进入正式 API 文档起草。
  - 仍未定死的字段必须继续保留“未决标记”，不能在评审中被默认视作已确认。
- 仍需带着责任人推进的项：
  - `expired` 阈值
  - 是否支持前台删除
  - 服务端是否需要草稿占位

### 4.5 后台边界结论

- 当前结论：`Go`
- 建议口径：
  - 后台默认只保留最小审核、规则、看板、敏感处置能力。
  - 敏感内容不进入普通后台可见范围。
  - 人工删除、权限控制、审计留痕是后台最小闭环的一部分，不是后补项。
