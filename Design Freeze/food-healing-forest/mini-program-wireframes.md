# 食愈森林小程序 MVP 低保真原型说明

## 1. 文档目标

- 固化 `MVP` 小程序关键页面的线框结构和跳转关系。
- 重点覆盖 `内容拉起 -> 发起记录 -> 饭后补感受 -> 完成反馈 -> 森林反馈` 主闭环。
- 只描述低保真结构，不展开视觉皮肤和品牌插画细节。

## 2. 全局导航

### 2.1 底部导航

- `晨光`
- `手帐`
- `森林`

### 2.2 为什么本轮不放 `食方`

- 版本规划已将 `食方` 放到 `V1`
- 当前 `MVP` 目标是验证记录闭环，不是扩展内容深度
- 如果此时保留第四个主入口，会稀释首页到记录动作的转化

## 3. 页面关系

```text
晨光首页
  -> 发起记录页
  -> 手帐列表

手帐列表
  -> 记录详情
  -> 饭后补感受页

提醒消息
  -> 饭后补感受页

饭后补感受页
  -> 完成反馈弹层
  -> 森林页

森林页
  -> 手帐列表
```

## 4. 页面 A1：晨光首页

### 4.1 页面目标

- 给用户一个每天愿意打开的轻理由
- 在不堆信息的情况下把用户带入记录动作

### 4.2 线框结构

```text
+--------------------------------------------------+
| 日期 / 问候语                                     |
|--------------------------------------------------|
| [日签卡片]                                        |
|  插画占位                                         |
|  一句治愈文案                                     |
|  [保存图片]                                       |
|--------------------------------------------------|
| [主按钮：记录这一餐]                              |
|--------------------------------------------------|
| 今日状态提示                                      |
|  - 若有待补记录：你还有 1 条饭后感受待补充        |
|  [去补感受]                                       |
|--------------------------------------------------|
| Tab: 晨光 | 手帐 | 森林                           |
+--------------------------------------------------+
```

### 4.3 交互说明

- 点击 `记录这一餐` 进入 `发起记录页`
- 若当天存在 `awaiting_post_meal_feedback` 记录，首页优先显示 `去补感受`
- `保存图片` 只做基础海报导出，不做复杂编辑

### 4.4 页面状态

- 正常态：日签正常展示
- 兜底态：日签缺失时显示默认文案卡片
- 待补态：存在未完成记录时显示醒目的待办提醒

## 5. 页面 A2：发起记录页

### 5.1 页面目标

- 让用户以最低成本完成一餐记录发起

### 5.2 线框结构

```text
+--------------------------------------------------+
| 返回    记录这一餐                                |
|--------------------------------------------------|
| [餐图上传框]                                      |
|  拍照 / 从相册选择                                |
|--------------------------------------------------|
| 现在的心情怎么样？                                |
| [开心] [平静] [焦虑] [疲惫] [低落] [还好]         |
|--------------------------------------------------|
| [提交本餐记录]                                    |
|--------------------------------------------------|
| 轻提示：不需要填写卡路里和克数                    |
+--------------------------------------------------+
```

### 5.3 交互说明

- 未上传图片也可讨论，但当前 PRD 默认以 `餐图 + 心情` 为提交结构
- 未选心情前，提交按钮不可用
- 提交成功后直接进入 `记录详情 / 待补状态`

### 5.4 页面状态

- 初始态：上传区为空
- 上传中：上传框显示进度和禁用态
- 上传失败：展示重试按钮，不丢失已选心情
- 提交成功：进入 `awaiting_post_meal_feedback`

## 6. 页面 A3：饭后补感受页

### 6.1 页面目标

- 把“饭后 2 小时补感受”做成一个轻动作，而不是二次填写负担

### 6.2 线框结构

```text
+--------------------------------------------------+
| 返回    饭后感受                                  |
|--------------------------------------------------|
| [本餐缩略图]                                      |
| 用餐时间：12:35                                   |
| 当时心情：平静                                    |
|--------------------------------------------------|
| 吃完这顿后，你现在感觉怎么样？                    |
| [轻盈] [满足] [腹胀] [犯困] [暖和] [没变化]       |
|--------------------------------------------------|
| [完成这条记录]                                    |
|--------------------------------------------------|
| 提示：如果错过提醒，你也可以随时回来补充          |
+--------------------------------------------------+
```

### 6.3 交互说明

- 从提醒进入和从手帐进入，页面结构一致
- 感受标签允许多选
- 提交成功后，记录状态进入 `completed`

### 6.4 页面状态

- 待补态：默认显示感受选择区
- 已补态：展示已完成摘要，不重复出现主提交按钮
- 过期提示态：若提醒过期，仅增加提示文案，不改变主页面结构

## 7. 页面 A4：手帐列表与记录详情

### 7.1 页面目标

- 让用户清楚知道自己记录了什么、哪些还没补完

### 7.2 列表页线框

```text
+--------------------------------------------------+
| 手帐                                              |
|--------------------------------------------------|
| 今天                                              |
| [记录卡片]                                        |
|  餐图缩略图 | 午餐 | 平静                         |
|  状态：待补饭后感受                               |
|  [去补感受]                                       |
|--------------------------------------------------|
| [记录卡片]                                        |
|  餐图缩略图 | 早餐 | 开心                         |
|  状态：已完成                                     |
|  [查看详情]                                       |
|--------------------------------------------------|
| 历史记录                                          |
| [按天分组的记录卡片列表]                          |
|--------------------------------------------------|
| Tab: 晨光 | 手帐 | 森林                           |
+--------------------------------------------------+
```

### 7.3 详情页结构

```text
+--------------------------------------------------+
| 返回    记录详情                                  |
|--------------------------------------------------|
| 餐图大图                                          |
| 用餐时间 / 当时心情 / 当前状态                    |
|--------------------------------------------------|
| 若待补：展示 [去补感受]                           |
| 若已完成：展示饭后感受标签与完成时间              |
|--------------------------------------------------|
| 轻反馈文案 / 小贴纸                               |
+--------------------------------------------------+
```

### 7.4 页面状态

- 空列表态：提示今天还没有记录，并给 `记录这一餐` 按钮
- 待补态：卡片优先置顶
- 已完成态：可查看完整内容

## 8. 页面 A5：森林页

### 8.1 页面目标

- 给用户“坚持记录有反馈”的感知
- 不把首版做成复杂养成游戏

### 8.2 线框结构

```text
+--------------------------------------------------+
| 森林                                              |
|--------------------------------------------------|
| [单株植物成长区]                                  |
|  植物插画占位                                     |
|  当前阶段：幼苗 / 生长中 / 舒展                    |
|--------------------------------------------------|
| 连续记录 4 天                                     |
| 本月完整记录 9 条                                 |
|--------------------------------------------------|
| 若有待补记录：                                    |
|  还有 1 条记录可以补完                            |
|  [去补感受]                                       |
|--------------------------------------------------|
| 轻反馈说明：每完成一条完整记录，植物会继续生长    |
|--------------------------------------------------|
| Tab: 晨光 | 手帐 | 森林                           |
+--------------------------------------------------+
```

### 8.3 页面状态

- 初始态：植物处于最小阶段
- 成长态：根据 `completed` 数量变化阶段
- 待补引导态：存在未完成记录时提供回补入口

## 9. 本轮原型评审重点

- 首页是否已经足够克制，没有把 V1 内容提前塞进来
- 发起记录是否真的比传统饮食记录更轻
- `待补感受` 是否在首页、手帐、森林三处都能回到闭环
- `森林` 是否承担反馈，不承担复杂玩法
- 是否能支撑 `提醒失败仍可手动回补` 这一核心降级逻辑

## 10. 当前建议

- 这套小程序原型先用于 `M1` 低保真评审
- 若 `M0` 最终确认仍允许 `食方` 保留在首版导航，再单独补一个导航变体，不要直接改动现有主方案
